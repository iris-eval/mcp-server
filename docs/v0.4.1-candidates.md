# v0.4.1 Candidates

> **Status:** Working list, no targeted ship date. v0.4.0 is the headline release; v0.4.1 will be a focused patch when material cause appears (a customer bug, a security finding, a small-but-clear quality win). Per `feedback_scope_discipline.md` we don't bundle hygiene into a release — each item lands when it's the right time.

## Surfaced gaps from v0.4 ship process

### 1. Tighten `since` parameter validation on `/api/v1/traces`
- **Source:** Track 6 dashboard walk findings (`strategy/proof/track6-dashboard-walk-findings-2026-04-24.md`).
- **Symptom:** `GET /api/v1/traces?since=not-a-date` returns 200 with the malformed value silently ignored. SQL layer is parameterized so injection isn't possible; the issue is sloppy contract.
- **Fix shape:** Zod-validate the `since` query param as an ISO-8601 timestamp; return 400 on invalid input with the same error envelope the rest of the API uses.
- **Effort:** ~30 min including tests. Touch surface: `dashboard/server.ts` route handler + new unit test in `tests/unit/api/`.
- **Severity:** v0.4.1 candidate (defense-in-depth, non-blocking). Client-side already validates; this is server-side hardening.

### 2. Surface `/api/v1/audit?action=*` enum values in error message
- **Source:** Track 6 dashboard walk findings.
- **Symptom:** `GET /api/v1/audit?action=deploy` returns 400 with the full enum list (`rule.deploy / rule.delete / rule.toggle / rule.update`). The error is correct but the namespace prefix is non-obvious to API consumers.
- **Fix shape:** API reference doc note + maybe a deprecation alias `action=deploy → action=rule.deploy` in the handler with a console warn.
- **Effort:** ~15 min docs only, ~1h with the alias.
- **Severity:** Cosmetic. Defer until someone hits it from a real integration.

### 3. Cosign verify in clean-dir verification scripts
- **Source:** `scripts/track9-rc-verify.sh` + `scripts/track10-v040-verify.sh` skip cosign when not installed locally.
- **Fix shape:** Either install cosign in CI verify lane and run `cosign verify` against the live registry image, or document the verification recipe in `docs/security.md` for end users.
- **Effort:** ~1h to wire into CI; ~15 min to add a docs section.
- **Severity:** v0.4.1+ — verification recipe in /security page would be customer-visible value.

## Future-work notes lifted from v0.4 dedicated docs

### From `docs/llm-as-judge.md`

- **Pricing table maintenance:** when Anthropic / OpenAI publish new models or change rates, update `src/eval/llm-judge/pricing.ts` + add a CHANGELOG note. Today's table is current as of 2026-04-24.
- **Streaming responses:** judge currently fetches full response. Streaming would let the judge cap output tokens early (cost optimization) and improve perceived latency on long rationales. Not blocking; nice-to-have for v0.5+.

### From `docs/semantic-citation-verify.md`

- **PDF support:** resolver currently rejects non-text content-types. Adding `pdf-parse` would extend the citation surface to academic PDFs. Heavy dep; opt-in vs default.
- **Bibliography parsing:** numbered `[N]` and author-year `(Smith, 2019)` citations resolve to "unresolvable_kind" because Iris can't see the bibliography section. A future extractor that parses a `## References` block and links inline numbered cites to bibliography entries would close that gap. ~1d effort; not v0.4.1 — a small feature.

### From `docs/otel-integration.md`

- **gRPC transport:** OTLP/HTTP JSON only in v0.4. Adding gRPC requires `@grpc/grpc-js` + `@opentelemetry/proto` (both heavy). Defer to v0.5 or skip entirely (collector front-ending is the standard pattern).
- **OpenTelemetry Logs / Metrics signals:** v0.4 only exports Traces. Adding Logs (audit log → OTel Logs) and Metrics (eval pass-rate → OTel Metric) would unlock dashboards in any OTel-aware backend. ~3d effort; v0.5 candidate.

### From `docs/api-reference.md`

- **OpenAPI schema:** the dashboard API is documented in markdown. A generated OpenAPI 3.0 schema would unlock client-codegen + Stoplight-style explorer + automatic SDK generation.
- **REST API stable contract:** dashboard API is currently `/api/v1/*` but versioning policy isn't formal. v0.5 should declare backward-compat guarantees or a deprecation policy.

## Triggers for v0.4.1 cut

A v0.4.1 patch ships when at least ONE of these conditions hits:
- A customer report of a real production bug.
- A security advisory affecting a v0.4.0 dependency (Dependabot HIGH+).
- A regression discovered in CI that wasn't caught by 372 tests but should have been.
- ≥3 of the above gaps becoming customer-blocking.

Otherwise the next ship is **v0.5** (Cloud Tier — see `docs/v0.5-cloud-tier-scope.md`).
